## 业务背景和价值

AI 生图系统已支持 LinkFox、自研、阿里绘蛙多模型并行。阿里绘蛙上线初期，其查询图片生成结果接口暂不返回图片宽高，系统因此将阿里绘蛙图片的宽高统一存为 `0`。

现阿里绘蛙查询结果接口已在 `imageResults[]` 中新增返回每张图片的 `width`、`height`，本期需同步解析并存储真实宽高，完善后续新生成图片的数据完整性。

## 需求概述

沿用现有阿里绘蛙图片结果查询及落库触发逻辑；当现有流程保存 `imageResults[].imageUrl` 时，同步读取同一结果对象中与其平级的整数类型字段 `width`、`height`，并写入数据库 `image` 图片表已有的同名字段。

本期不新增数据库字段、不调整前端页面，也不对历史宽高为 `0` 的数据执行专项修复。

## 功能需求

### 1. 图片宽高解析与存储

阿里绘蛙查询结果中的单张图片结构如下：

```json
{
  "imageUrl": "https://img.alicdn.com/example.jpg",
  "width": 2048,
  "height": 2048
}
```

处理规则：

1. 沿用现有阿里绘蛙图片结果查询、状态判断和图片落库逻辑，不新增或改变任务处理状态。
2. 遍历 `imageResults[]`，将同一结果对象中的 `imageUrl`、`width`、`height` 作为同一张图片的结果数据。
3. 将 `imageResults[].width`、`imageResults[].height` 分别写入数据库 `image` 图片表已有的 `width`、`height` 字段。
4. 每个 `imageResults[]` 图片结果沿用现有图片落库逻辑关联一条 `image` 图片记录，不改变现有图片关联关系。
5. 不通过外层 `imageUrls[]` 的数组下标与 `imageResults[]` 强行对应。
6. LinkFox、自研模型现有图片宽高处理逻辑保持不变。

### 2. 异常兼容

1. `width` 或 `height` 未返回、为 `null`、非法值或小于等于 `0` 时，对应字段按 `0` 处理。
2. 宽高异常不得影响 `imageUrl`、任务状态及其他图片结果正常落库；单张图片宽高异常不影响同一任务中的其他图片。
3. 数据库已有合法正整数宽高时，在现有任务结果处理流程中，后续空值、非法值或 `0` 不得覆盖原有合法值。
4. 数据库当前宽高为 `0` 时，如现有任务结果处理流程后续获得合法正整数，可更新为真实宽高。

### 3. 影响范围

本需求统一作用于阿里绘蛙查询结果及图片落库的共享后端链路，覆盖：

- 美工版阿里绘蛙生图任务；
- 运营版阿里绘蛙生图任务；
- 单个新建、批量创建、复制、批量复制、重跑等能够产生新阿里绘蛙图片结果的入口。

各入口无需分别开发前端逻辑，但需回归新生成图片的结果保存、查看和审核链路。

### 4. 历史数据处理

1. 本期不执行 SQL、批处理或重新调用阿里绘蛙接口，专项修复历史宽高为 `0` 的图片数据。
2. 本需求上线后新产生或按现有任务结果处理流程重新处理的阿里绘蛙图片结果，按本期新口径保存宽高。
3. 不为补齐历史宽高新增查询任务或扩大现有查询范围。

## 验收标准

1. 使用阿里绘蛙真实成功响应完成联调，确认 `width`、`height` 位于 `imageResults[]` 内、与 `imageUrl` 平级且为整数类型。
2. 美工版、运营版分别创建阿里绘蛙任务，生成图片落库后，每张图片的 `imageResults[].width`、`imageResults[].height` 均正确写入 `image` 表对应字段。
3. 同一任务包含多张图片时，各图片按自身 `imageResults[]` 对象分别保存宽高，不出现图片与宽高错配。
4. 单个新建、批量创建、复制、批量复制、重跑产生的新阿里绘蛙图片均按新口径保存真实宽高。
5. 宽高未返回、为 `null`、非法值、`0` 或负数时，对应字段按 `0` 处理，图片及其他结果仍正常落库。
6. 已有合法宽高不被后续空值、非法值或 `0` 覆盖；已有值为 `0` 时，可在现有任务结果处理流程中被后续合法正整数补齐。
7. 上线前已落库且宽高为 `0` 的历史数据不执行专项修复。
8. LinkFox、自研模型现有图片宽高存储不受影响；共享图片查看、审核链路原有操作正常。
