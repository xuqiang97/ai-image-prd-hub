## 业务背景和价值

AI 生图系统已支持 LinkFox、自研、阿里绘蛙多模型并行。阿里绘蛙上线初期，其查询图片生成结果接口暂不返回图片宽高，系统因此将阿里绘蛙图片的宽高统一存为 `0`。

现阿里绘蛙查询结果接口已在 `imageResults[]` 中新增返回每张图片的 `width`、`height`，本期需同步解析并存储真实宽高，完善后续新生成图片的数据完整性。

## 需求概述

阿里绘蛙图片生成成功后，系统遍历查询结果中的 `imageResults[]`，读取与 `imageUrl` 平级的 `width`、`height`，并写入数据库 `image` 图片表已有的 `width`、`height` 字段。

本期不新增数据库字段、不调整前端页面，也不对历史宽高为 `0` 的数据执行专项修复。

## 功能需求

### 1. 图片宽高解析与存储

阿里绘蛙查询结果中的单张图片结构如下：

```json
{
  "imageUrl": "https://img.alicdn.com/example.jpg",
  "width": 2048,
  "height": 2048
}
```

处理规则：

1. 遍历 `imageResults[]`，以每个结果对象中的 `imageUrl`、`width`、`height` 作为同一张图片的结果数据。
2. 将 `imageResults[].width` 写入数据库 `image` 图片表已有的 `width` 字段。
3. 将 `imageResults[].height` 写入数据库 `image` 图片表已有的 `height` 字段。
4. 不通过外层 `imageUrls[]` 的数组下标与 `imageResults[]` 强行对应。
5. LinkFox、自研模型现有图片宽高处理逻辑保持不变。

### 2. 异常兼容

1. `width` 或 `height` 未返回、为 `null`、非法值或小于等于 `0` 时，对应字段按 `0` 存储。
2. 宽高异常不得影响 `imageUrl`、任务状态及其他图片结果正常落库。
3. 单张图片宽高异常，不影响同一任务中其他图片结果处理。
4. 数据库已有合法正整数宽高时，后续重复处理返回的空值、非法值或 `0` 不得覆盖原有合法值。
5. 数据库当前宽高为 `0`，后续正常处理获得合法正整数时，允许更新为真实宽高。

### 3. 影响范围

本需求统一作用于阿里绘蛙查询结果及图片落库的共享后端链路，覆盖：

- 美工版阿里绘蛙生图任务；
- 运营版阿里绘蛙生图任务；
- 单个新建、批量创建、复制、批量复制、重跑等能够产生新阿里绘蛙图片结果的入口。

各入口无需分别开发前端逻辑，但需回归新生成图片的结果保存、查看和审核链路。

### 4. 历史数据处理

1. 本期不执行 SQL、批处理或重新调用阿里绘蛙接口，专项修复历史宽高为 `0` 的图片数据。
2. 本需求上线后新产生或按现有正常机制重新处理的阿里绘蛙图片结果，按本期新口径保存宽高。
3. 不为补齐历史宽高新增查询任务或扩大现有查询范围。

## 风险描述

阿里绘蛙正式环境若仍存在部分结果不返回宽高的情况，系统按 `0` 兼容，不阻断图片结果落库。上线前需使用真实成功结果确认 `width`、`height` 的字段类型和响应层级与接口文档一致。

## 验收标准

1. 美工版、运营版分别创建阿里绘蛙任务，生成成功后，每张图片的 `imageResults[].width`、`imageResults[].height` 均正确写入 `image` 表对应字段。
2. 同一任务包含多张图片时，各图片按自身 `imageResults[]` 对象分别保存宽高，不出现图片与宽高错配。
3. 单个新建、批量创建、复制、批量复制、重跑产生的新阿里绘蛙图片均按新口径保存真实宽高。
4. 宽高未返回、为 `null`、非法值、`0` 或负数时，对应字段保存为 `0`，图片及其他结果仍正常落库。
5. 已有合法宽高不被后续空值、非法值或 `0` 覆盖；已有值为 `0` 时，可被后续合法正整数补齐。
6. 上线前已落库且宽高为 `0` 的历史数据不执行专项修复。
7. LinkFox、自研模型现有图片宽高存储不受影响。
8. L1 任务列表、L2 审核工作台、L3 沉浸式大屏中，阿里绘蛙新生成图片可正常查看和审核，原有操作不受影响。
