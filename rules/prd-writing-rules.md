# AI 生图小需求 PRD 写作规则

本文件用于沉淀 AI 生图轻量小需求 PRD 的统一写法，后续给 AI 或 Codex 参考时可直接引用。

## 1. 文档交付原则

### 1.1 只维护一份正式文档

每个需求只维护一份 `README.md`，该文件即为唯一正式 PRD。

不再拆分：

- GitHub README.md 实施版 PRD
- 钉钉模板版 PRD

统一改为：

- README.md = 唯一正式需求文档 = 可复制到钉钉流程的 PRD

### 1.2 前半部分遵循公司模板

PRD 前半部分尽量遵循公司钉钉模板格式，包括：

- 标题
- 修改纪要
- 调研纪要
- 业务背景和价值
- 需求概述
- 功能需求
- 风险描述
- 验收标准
- 服务验收
- 初始化用户权限
- 其他

后半部分不需要机械硬套模板，可根据需求实际情况灵活展开。

### 1.3 以准确落地为目标

PRD 应同时满足：

- 研发能准确实现
- 测试能准确验收
- 业务/领导能看懂价值和边界
- 钉钉流程能直接复制使用
- 后续 AI 能基于历史文档继续复用经验

## 2. 仓库收纳规则

### 2.1 适合放入本仓库的需求

适合放入 `ai-image-prd-hub` 的需求，是 AI 生图项目中不需要单独建立 HTML 原型仓库的轻量需求，例如：

- 取数逻辑优化
- 接口字段调整
- 配置项默认值
- 前端 localStorage 本地缓存
- 状态流转规则
- 定时任务
- 数据同步
- 后端处理口径
- 线上问题修复
- 不需要页面交互确认的小需求

### 2.2 不适合放入本仓库的需求

如需求涉及以下内容，建议单独建立原型仓库：

- 新增页面
- 页面改版
- 新增复杂弹窗
- 列表字段和筛选项有明显页面变化
- 多页面流程
- 复杂状态流转可视化
- 权限矩阵可视化
- BI 看板
- 需要业务方先确认页面效果的交互需求

判断标准不是“前端 / 后端”，而是：

**是否需要单独 HTML 原型确认。**

例如：前端 localStorage 本地缓存属于前端实现，但如果页面展示和交互不变化，就可以放入本仓库。

## 3. 写作风格

### 3.1 小需求要轻量

日常小需求不要写成重 PRD，重点写清：

- 问题背景
- 需求范围
- 核心规则
- 禁止误解点
- 风险点
- 验收标准

### 3.2 避免模糊表达

尽量避免使用以下模糊表达：

- 尽量
- 可能
- 视情况
- 相关逻辑
- 保持一致
- 正常处理

应改为明确规则，例如：

- 默认值读取浏览器 localStorage 中上一次成功保存的值。
- 若 localStorage 中无缓存值，则使用系统默认值。
- 当前需求不涉及历史任务数据变更。

### 3.3 修改纪要保持精炼

最终版本一般只保留 V1.0 即可，不需要记录多轮讨论过程。

示例：

| 日期 | 版本 | 修改人 | 主要修改点 |
|---|---|---|---|
| 2026-07-09 | V1.0 | 徐强 | 新建 PRD，明确本次配置项默认值缓存规则、影响范围和验收标准。 |

### 3.4 调研纪要只写有效信息

调研纪要用于记录业务侧或关键干系人的输入。

如果没有明确调研对象，可以保留表格并写“暂无”。

## 4. 常见需求类型写法

### 4.1 localStorage 本地缓存类

需要写清：

- 缓存字段名称或字段含义
- 写入时机
- 读取时机
- 默认值兜底规则
- 浏览器维度限制
- 账号维度是否隔离
- 清缓存后的表现
- 是否影响历史任务

推荐叫法：

**前端 localStorage 本地缓存**

### 4.2 取数逻辑优化类

需要写清：

- 触发入口
- 查询参数
- 数据源
- 优先级规则
- 多条命中时取哪一条
- 无数据时如何处理
- 单个导入和批量导入是否统一
- 历史逻辑是否保留

### 4.3 接口字段调整类

需要写清：

- 新增 / 调整 / 废弃字段
- 字段类型
- 是否必填
- 默认值
- 前端传参规则
- 后端兼容规则
- 老数据处理方式

### 4.4 线上问题修复类

需要写清：

- 线上问题表现
- 影响范围
- 根因或疑似根因
- 修复口径
- 不影响范围
- 验收样例
- 是否需要补数据

线上问题转需求时，不要写重 PRD，重点保证研发和测试对修复口径无歧义。

## 5. 目录和索引维护规则

### 5.1 新需求目录

新增需求时，在对应月份下建立目录：

```text
prd/YYYY-MM/M-禅道号-英文短名/README.md
```

暂无禅道号时：

```text
prd/YYYY-MM/TEMP-YYYYMMDD-英文短名/README.md
```

### 5.2 同步维护索引

新增需求后，需要同步更新：

1. 仓库顶层 `README.md` 的需求索引。
2. 对应月份 `prd/YYYY-MM/README.md` 的本月需求索引。

### 5.3 状态流转

建议使用以下状态：

- 草稿
- 待评审
- 已确认
- 开发中
- 已上线
- 已废弃

## 6. 后续给 AI 的推荐提示词

可以直接使用以下提示词：

> 请参考这个仓库的 PRD 写作规则和模板：`ai-image-prd-hub`。本次需求不需要 HTML 原型，请在 `prd/YYYY-MM/需求目录/README.md` 下新增唯一一份 README.md PRD，并同步更新顶层索引和月份索引。

如本次需求涉及页面交互，则应先判断是否需要单独建立 HTML 原型仓库，不要直接塞进本 PRD Hub。
