# AGENTS.md

本文件用于指导 ChatGPT、Codex 及其他 AI 在本仓库中新增和维护 AI 生图轻量需求 PRD。

## 1. 唯一文档原则

每个需求最终只维护一份：

```text
prd/YYYY-MM/需求目录/README.md
```

该 README.md 同时用于 GitHub 沉淀、研发实施、测试验收、钉钉流程和后续 AI 复用。

不得额外生成钉钉版、GitHub 实施版、精简版、月份汇总版或其他重复文档。

## 2. 仓库定位

本仓库只收录**不需要单独制作 HTML 原型**的 AI 生图轻量需求。

判断标准：

> 本需求是否需要先通过页面原型确认交互和展示效果。

需要原型时，应单独建立原型仓库，通过 `HTML 原型 + 唯一 README.md PRD` 交付。

适合本仓库的需求包括：取数逻辑、接口字段、已上线接口的兼容扩展、配置规则、状态流转、localStorage、本地缓存、数据同步、后端处理口径、定时任务、线上修复和页面无明显变化的轻量前端逻辑。

## 3. 新需求处理流程

1. 判断是否需要 HTML 原型。
2. 梳理需求边界、是否拆分和关键缺失信息。
3. 判断更新历史需求还是新建独立需求。
4. 在聊天中完成需求澄清、实现口径和验收标准确认。
5. 等用户提供正式需求名称、禅道编号和禅道链接。
6. 信息齐全后，直接创建 `M-禅道号-英文短名` 正式目录。
7. 生成或更新该需求唯一的 README.md。
8. 新建需求时，更新根目录 README.md 的唯一需求索引。

默认规则：

- 正式需求名称、禅道编号和链接齐全前，只继续讨论，不提前写入 GitHub。
- 默认不创建 `TEMP` 临时目录，避免后续删除、迁移和重命名。
- 仅在用户明确要求提前落库时创建临时目录，并说明后续需要正式化。
- 不创建月份 README，不维护第二份索引。

## 4. 更新原需求还是新建需求

### 4.1 更新原 README.md

以下情况更新原需求，不新建目录：

- 同一禅道需求评审前继续补充或修正规则。
- 研发、测试提出澄清，但仍属于同一实施范围。
- 修改内容没有形成独立开发、测试或上线对象。

更新时保留原目录和禅道编号，仅修改发生变化的内容，不得无依据改写已确认口径。

### 4.2 新建 README.md

以下情况新建独立需求：

- 已有新的独立禅道需求。
- 原需求已上线，本次属于后续迭代。
- 新增内容可以独立开发、测试或上线。
- 本次存在新的需求边界和验收标准。
- 线上问题被单独转为修复需求。

不得把多个可独立实施的需求长期叠加在同一 README.md 中。

同一批次、同一页面或共享组件内的轻量顺带优化，如果业务目标相关、由同一批研发实现、无需独立测试或上线，可以作为同一需求的次要范围；但必须在需求概述、影响入口和验收标准中单独写清。能够独立开发、测试或上线，或具有独立业务目标时，仍应拆分为独立需求。

## 5. 目录规则

```text
prd/YYYY-MM/M-禅道号-英文短名/README.md
```

示例：

```text
prd/2026-07/M-36422-model-quantity-cache/README.md
prd/2026-06/M-35944-asin-query-logic/README.md
```

英文短名使用小写字母和中划线，建议 3～6 个单词，优先描述需求核心，不使用无意义的 `update`、`change`、`optimize` 单独命名。

## 6. README.md PRD 写作规则

### 6.1 基础结构

PRD 前部一般包括：

1. 文档标题
2. 禅道需求完整 URL
3. GitHub PRD 完整 URL
4. 修改纪要
5. 调研纪要
6. 业务背景和价值
7. 需求概述
8. 功能需求
9. 风险描述
10. 验收标准

标题层级：

- 需求总标题使用一级标题 `#`。
- 主要章节使用二级标题 `##`。
- 章节下的模块和规则使用三级标题 `###`。
- 不使用加粗正文模拟标题。

禅道需求和 GitHub PRD 必须直接展示完整 URL，不使用 `[文字](URL)` 形式隐藏地址。GitHub PRD 放在禅道需求下一行，方便复制到钉钉文档后直接查看和回跳原文。

推荐起始结构：

```text
# M-xxxxx【AI生图】[运营/美工/美工+运营/公共]需求名称（责任人：徐强）

禅道需求：https://pm.zhcxkj.com/zentao/story-view-xxxxx.html

GitHub PRD：https://github.com/xuqiang97/ai-image-prd-hub/blob/main/prd/YYYY-MM/M-xxxxx-英文短名/README.md

## 修改纪要

| 日期 | 版本 | 修改人 | 主要修改点 |
|---|---|---|---|
| YYYY-MM-DD | V1.0 | 徐强 | 新建 PRD，明确本次需求范围、核心规则和验收标准。 |

## 调研纪要

| 日期 | 业务部门 | 角色 | 调研对象（干系人） | 调研内容和结论 |
|---|---|---|---|---|
| YYYY-MM-DD | 暂无 | 暂无 | 暂无 | 暂无 |

## 业务背景和价值

## 需求概述

## 功能需求

### 1. 具体模块或规则

## 风险描述

## 验收标准
```

### 6.2 轻量与聚焦原则

轻量需求应**简明扼要、突出本期实现内容**，但不得为了缩短篇幅丢失会影响产品判断、研发实现或测试验收的必要规则。

写作时优先聚焦：

- 业务为什么提出该需求。
- 本期具体修改哪些入口、模块和逻辑。
- 核心业务规则、处理优先级和成功条件。
- 必要的异常处理、边界条件和兼容规则。
- 可以直接执行和验证的验收标准。

组织内容时：

- 同一规则只在最合适的章节完整说明，避免在功能需求、异常和边界、风险描述、验收标准中反复展开。
- “不在本期范围”只保留容易被研发、测试误解，或确实会影响实施边界的事项，不为形式完整罗列大量无关内容。
- 异常和边界优先并入对应功能规则；只有跨多个模块、无法归入具体功能的异常才单独成章。
- 验收标准用于验证功能规则，可以合并相近场景、适当控制篇幅，不机械复制前文全部内容。
- 业务背景优先描述当前状态、业务影响和本期目标。除非问题根因会直接影响修复范围或验收口径，否则不使用“遗漏”“漏做”“做错”等带有归责意味的历史表述，也不写与本期实现无关的人员或团队更替。
- 不适用章节直接删除，不机械套模板。
- 明确区分已确认前提、上线条件和未关闭风险：已由业务、上下游或现网验证确认的事实，直接写入业务背景或功能规则；确定必须满足的发布顺序、检查项写入对应功能规则或验收标准；只有仍有发生可能且会影响实现、上线或验收的事项才保留在风险描述中，已关闭事项不重复作为风险。

验收标准必须覆盖：

- 核心正常链路。
- 关键失败场景。
- 业务端隔离和账号隔离。
- 历史兼容和配置变化。
- 容易引发实现偏差的特殊场景。

避免模糊表达，如“尽量”“可能”“视情况”“正常处理”“相关逻辑”“保持一致”，应改为可执行、可测试的明确规则。

### 6.3 修改纪要

- 同一需求在正式提交研发、测试评审前，无论经历多少轮内部讨论、自查或优化，也无论是否已经写入 GitHub，均继续维护 `V1.0`；修改纪要不记录 AI 与用户的内部讨论过程，GitHub 提交次数不等同于文档版本升级。
- 正式提交评审或形成对外基线后，发生影响需求范围、业务规则、接口口径或验收标准的实质变化时，才增加 `V1.1`、`V1.2`。
- 仅修正文案、格式、错别字或压缩重复内容，且不影响研发实现和测试验收时，不机械新增版本。

### 6.4 调研纪要

- 调研纪要原则上记录真实业务调研和业务干系人结论，优先写明业务部门、使用角色、实际问题和确认口径。
- 业务方提出的问题、实际使用反馈和确认口径均可作为调研纪要。
- 研发、测试、产品内部的口头同步、任务交接或“已经告知研发”等信息，不作为业务调研纪要。
- 研发或测试提出的技术可行性、接口约束、实现风险等结论，如确实影响产品决策，应写入对应功能规则、风险描述或上线协作说明，不冒充业务调研。
- 已有美工、运营等业务人员的真实反馈时，不得仍填写“暂无”。
- 没有调研信息时才写“暂无”，不得虚构。

M-36422 调研结论参考表达：

> 美工和运营业务反馈，用户实际使用时通常不会每次都按系统允许的最大张数生图，导致新增任务或批量导入时需要反复调整“模型与数量配置”，因此希望系统记忆用户个人上一次成功使用的配置。

### 6.5 初稿后的主动复核

AI 完成 PRD 初稿后，必须主动进行一次全盘复核，再交付最终稿。重点检查：

- 是否存在规则冲突或前后矛盾。
- 是否遗漏真实业务入口；同一组件被多个页面、路由或业务端复用时，是否逐一列明实际入口，而不是只写“相关页面”“对应入口”。
- 主需求之外的轻量顺带优化，是否也在需求概述、影响范围和验收标准中明确体现。
- 是否存在研发可能产生不同理解的歧义。
- 是否准确区分现有字段与新增字段、请求入参与响应出参，以及名称相近的单数/复数字段。
- 已上线接口是否覆盖现网调用兼容、独立发布、下游沟通和接口文档更新。
- 是否为了严谨而写得过重。
- 是否可以通过合并重复内容提升可读性。
- 验收标准是否覆盖核心链路、失败场景、隔离、兼容和特殊场景。

最终目标是形成一份**篇幅适中、重点突出、口径明确、研发可执行、测试可验收**的唯一正式 README.md PRD。

### 6.6 通用经验沉淀

- 需求完成后，只将能够跨需求稳定复用的仓库流程、文档职责、写作规则和复核标准沉淀到根目录 `README.md` 或 `AGENTS.md`。
- 单个需求的业务数值、专属文案、页面入口、状态流转、上下游特殊约束和验收样例，只保留在该需求 README.md 中，不上升为仓库通用规则。
- 沉淀前先检查现有规则是否已经覆盖；优先精确补充或改写原条目，不重复新增近义规范。

## 7. 关联需求与参考资料

需求依赖历史 PRD、原型或业务资料时，按需增加：

```text
## 关联需求与参考资料

- 前置需求：M-xxxxx【需求名称】
- 历史 PRD：相对路径或 GitHub 链接
- 参考原型：GitHub 仓库或页面地址
- 影响模块：具体业务端、页面、接口或任务链路
```

优先使用仓库内相对路径；外部原型、禅道、语雀或接口文档可使用完整链接。只添加与本次实施直接相关的资料。

## 8. 常见需求重点

### 8.1 localStorage / 本地缓存

写清：缓存内容、读取时机、写入时机、默认值、账号和业务端隔离、浏览器/设备范围、清缓存表现、历史任务影响及异常兜底。

### 8.2 取数逻辑优化

写清：触发入口、查询参数、数据源、优先级、多条命中规则、无数据处理、单个/批量是否统一和历史逻辑影响。

### 8.3 接口字段调整

先通过接口文档、实际请求和实际响应还原现状，不得仅根据字段名称猜测业务含义。写清：

- 字段位于请求还是响应、所在层级、名称和类型。
- 字段是现有、新增、修改还是废弃，是否必填及默认值。
- `null`、空字符串、空数组、非法值和多参数同时传入时的处理。
- 参数组合对应的业务模式、处理优先级，以及是否允许自动降级或扩大查询范围。
- 现有字段和返回结构的兼容、历史数据处理及下游使用方式。
- 名称相近字段必须明确区分，特别是请求/响应、单数/复数和大小写差异。

请求和响应结构容易产生理解偏差时，应提供精简 JSON 示例，并注明示例对应的查询模式或业务场景。

### 8.4 已上线接口的兼容扩展

接口已被其他系统在线使用时，PRD 必须额外写清：

- 当前调用方、现有请求方式和线上使用场景，不能只描述改造后的新能力。
- 是否允许接口提供方先上线、调用方后续再逐步接入，以及旧调用方无需改造即可继续使用的兼容要求。
- 现有字段由必填改为非必填、新增入参或新增出参时，不得删除、重命名或改变现有字段类型和原有非空场景逻辑。
- 按参数组合识别模式时，写清优先级、空值处理和无结果行为；除非业务明确要求，不得静默自动扩大查询范围。
- 新增响应字段后，应使用当前线上调用方版本完成真实兼容验证，关注严格反序列化、字段校验和前端解析。
- 接口交付前同步调用方负责人，并更新 ShowDoc、Swagger 或其他正式接口文档，补充字段定义、请求/响应示例和参数优先级。
- 引用 CURL 或请求示例时，必须移除或脱敏真实 `Authorization`、Token、Cookie、密钥及其他敏感信息，禁止写入仓库。

### 8.5 线上问题修复

写清：问题表现、影响范围、修复口径、不影响范围、是否补数据和验收样例。修复型需求不要写成重 PRD。

### 8.6 轻量展示补齐与样式优化

写清：实际影响的业务端和全部页面入口、需要对齐的现有页面或组件、展示字段的数据来源、空值和超长处理、是否只改前端展示、共享组件的回归范围，以及原有操作不受影响的验收要求。

只是在现有页面补充已有字段、对齐另一业务端的既有展示，或调整局部间距、换行和布局时，通常不需要单独制作 HTML 原型；如涉及新的信息层级、复杂交互或整体页面改版，则应单独建立原型仓库。

## 9. 索引维护

只维护根目录 README.md 中的需求索引。

新建需求时补充：日期、需求编号、业务端、需求名称、需求类型和 PRD 链接。

索引按 PRD 创建日期倒序排列；开发、测试和上线状态以禅道为准，不在 GitHub 重复维护。更新已有需求名称或路径时，修正原索引行，不重复新增。

索引中的“业务端”按整份 PRD 的实际影响范围填写，不只照抄需求标题中的业务端标签；同一需求实际同时影响美工版和运营版时，填写“美工+运营”。

业务端建议：`运营`、`美工`、`美工+运营`、`公共`。

需求类型建议：`配置`、`接口`、`规则`、`修复`、`缓存`、`数据`、`其他`。

## 10. 禁止事项

- 不得为同一需求维护两份 PRD。
- 不得创建月份 README 或第二层索引。
- 不得额外创建重复模板文档。
- 不得为了套模板填写无意义章节。
- 不得擅自脑补业务规则、接口字段或验收口径。
- 不得修改历史需求的已确认口径，除非用户明确要求。
- 不得将明显需要 HTML 原型的需求强行放入本仓库。
- 不得用 GitHub 索引重复维护禅道中的开发、测试和上线状态。
- 不得将 PRD 顶部的禅道需求和 GitHub PRD 写成隐藏完整地址的 Markdown 链接。
- 不得在仓库文档中保留真实 Authorization、Token、Cookie、密钥或其他敏感信息。
- 未取得正式需求名称和禅道信息前，不得默认创建 `TEMP` 目录。
