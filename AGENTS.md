# AGENTS.md

本文件用于指导 ChatGPT、Codex 及其他 AI 在本仓库中新增和维护 AI 生图轻量需求 PRD。

## 1. 唯一文档原则

每个需求最终只维护一份：

```text
prd/YYYY-MM/需求目录/README.md
```

这份 `README.md` 同时承担以下用途：

- GitHub 需求沉淀
- 研发实施参考
- 测试验收参考
- 复制到钉钉需求流程
- 后续 AI 读取和复用

不得再额外生成：

- 钉钉版 PRD
- GitHub 实施版 PRD
- 精简版 PRD
- 月份汇总版 PRD
- 与需求 README.md 内容重复的模板副本

## 2. 仓库定位

本仓库只收录**不需要单独制作 HTML 原型**的 AI 生图轻量需求。

判断标准：

> 本需求是否需要先通过页面原型确认交互和展示效果。

如答案为“需要”，则不要将需求直接放入本仓库，应建议单独建立原型仓库，通过 `HTML 原型 + 唯一 README.md PRD` 交付。

适合收录的需求包括但不限于：

- 取数逻辑优化
- 接口字段调整
- 配置项规则
- 状态流转
- 前端 localStorage 等本地缓存规则
- 数据同步
- 后端处理口径
- 定时任务
- 线上问题修复
- 页面无明显变化的轻量前端逻辑

## 3. 新需求处理流程

收到新需求后，按以下顺序处理：

1. 判断是否需要 HTML 原型。
2. 梳理需求边界、是否拆分和关键缺失信息。
3. 判断应更新历史需求，还是新建独立需求。
4. 在对应月份下新建或更新需求目录。
5. 直接生成或更新该需求唯一的 `README.md` PRD。
6. 新建需求时，更新仓库根目录 `README.md` 的唯一需求索引。
7. 不创建月份 README，不维护第二份索引，不额外生成钉钉版 PRD。

## 4. 更新原需求还是新建需求

### 4.1 更新原 README.md

满足以下情况时，直接更新原需求 README.md，不新建目录：

- 同一个禅道需求尚未进入独立开发，继续补充或修正需求。
- 评审前调整需求范围、规则、字段或验收口径。
- 研发或测试发现原文档描述不清，需要补充澄清。
- 修改内容本质仍属于同一个实施范围，没有形成独立验收对象。

更新原 README.md 时：

- 保留原需求目录和禅道编号。
- 仅修改发生变化的内容。
- 修改纪要通常仍保持精炼；如已正式评审后发生实质变化，可新增版本记录。
- 不得无依据改写已经确认且未被用户要求调整的历史口径。

### 4.2 新建需求 README.md

满足以下任一情况时，应新建需求目录和 README.md：

- 已有新的独立禅道需求。
- 原需求已经上线，本次属于后续迭代。
- 新增内容可以独立开发、测试或上线。
- 本次存在新的需求边界和验收标准。
- 原需求只作为背景参考，本次实现目标已经不同。
- 线上问题被单独转为修复需求。

不得为了省事，把多个可独立实施的需求长期叠加到同一 README.md 中。

## 5. 目录规则

正式需求：

```text
prd/YYYY-MM/M-禅道号-英文短名/README.md
```

暂无禅道号：

```text
prd/YYYY-MM/TEMP-YYYYMMDD-英文短名/README.md
```

英文短名规则：

- 使用小写字母和中划线。
- 建议 3～6 个单词。
- 优先描述需求核心，不使用无意义的 `update`、`change`、`optimize` 单独命名。
- 禅道号确认后，可将 `TEMP` 目录调整为正式目录名，并同步更新顶层索引链接。

## 6. README.md PRD 写作规则

### 6.1 前半部分遵循公司模板

每个需求的 README.md 应直接按照公司钉钉模板版 PRD 风格编写，前半部分一般包括：

1. 文档标题
2. 禅道需求链接
3. 修改纪要
4. 调研纪要
5. 业务背景和价值
6. 需求概述
7. 功能需求
8. 风险描述
9. 验收标准

推荐起始结构：

```text
**M-xxxxx【AI生图】[运营/美工/公共]需求名称（责任人：徐强）**

禅道需求：[M-xxxxx](禅道链接)

# 修改纪要

| 日期 | 版本 | 修改人 | 主要修改点 |
|---|---|---|---|
| YYYY-MM-DD | V1.0 | 徐强 | 新建 PRD，明确本次需求范围、核心规则和验收标准。 |

# 调研纪要

| 日期 | 业务部门 | 角色 | 调研对象（干系人） | 调研内容和结论 |
|---|---|---|---|---|
| YYYY-MM-DD | 暂无 | 暂无 | 暂无 | 暂无 |

# 业务背景和价值

# 需求概述

# 功能需求

# 风险描述

# 验收标准
```

这只是 README.md 的写作结构要求，不代表需要额外创建一份模板文档。

### 6.2 后半部分按需展开

后半部分根据本次需求实际情况灵活组织，不机械填写所有章节。

可按需增加：

- 数据和接口口径
- 字段定义
- 取数优先级
- 缓存规则
- 状态流转
- 异常和边界
- 历史数据处理
- 服务验收
- 初始化权限
- 测试样例
- 回归范围
- 关联需求与参考资料

不适用的章节直接删除，不写空洞内容。

### 6.3 小需求写法

小需求应保持轻量，重点写清：

- 为什么要做
- 本期具体改什么
- 哪些范围不改
- 规则和优先级
- 异常情况如何处理
- 如何验收

避免模糊表达，例如：

- 尽量
- 可能
- 视情况
- 正常处理
- 相关逻辑
- 保持一致

应改为可执行、可测试的明确规则。

### 6.4 修改纪要

- 最终 PRD 通常只保留 `V1.0`，不记录 AI 与用户之间的多轮讨论过程。
- 如需求已经正式确认后又发生实质范围变化，可增加 `V1.1`、`V1.2` 等版本。
- 仅修正文案、错别字或格式时，不必机械新增版本。

### 6.5 调研纪要

只记录有效的业务调研和关键干系人结论。没有调研信息时，可写“暂无”，不得虚构调研对象或结论。

## 7. 关联需求与参考资料

新需求依赖历史需求、历史原型或其他业务资料时，应在该需求唯一 README.md 中按需增加：

```text
# 关联需求与参考资料

- 前置需求：M-xxxxx【需求名称】
- 历史 PRD：相对路径或 GitHub 链接
- 参考原型：GitHub 仓库或页面地址
- 影响模块：具体业务端、页面、接口或任务链路
```

规则：

- 优先使用仓库内相对路径链接历史 PRD，便于仓库迁移后继续访问。
- 外部原型仓库、禅道、语雀或接口文档可以使用完整链接。
- 仅添加与本次实施真正相关的资料，不堆积无关链接。
- 历史需求是参考依据，不代表本次自动继承其全部范围；本次 PRD 仍需明确自己的边界和验收标准。

## 8. 常见需求重点

### 8.1 localStorage / 本地缓存

写清：

- 缓存内容
- 写入时机
- 读取时机
- 默认值
- 账号是否隔离
- 浏览器和设备维度
- 清缓存后的表现
- 是否影响历史任务

### 8.2 取数逻辑优化

写清：

- 触发入口
- 查询参数
- 数据源
- 优先级
- 多条命中规则
- 无数据处理
- 单个和批量入口是否统一
- 历史逻辑影响

### 8.3 接口字段调整

写清：

- 字段名称和类型
- 新增、修改或废弃
- 是否必填
- 默认值
- 前端传参
- 后端兼容
- 历史数据处理

### 8.4 线上问题修复

写清：

- 问题表现
- 影响范围
- 修复口径
- 不影响范围
- 是否补数据
- 验收样例

修复型需求不要写成重 PRD。

## 9. 索引维护

只维护根目录 `README.md` 中的需求索引。

新建需求时补充：

- 日期
- 需求编号
- 业务端
- 需求名称
- 需求类型
- PRD 链接

索引规则：

- 按 PRD 创建日期倒序排列，最新需求放在最上方。
- 不保留空占位数据行。
- 项目开发、测试和上线状态以禅道为准，GitHub 索引不重复维护。
- 更新已有需求 README.md 时，一般不重复新增索引行；如需求名称或路径发生变化，同步修正原索引行。

业务端建议：`运营`、`美工`、`公共`。

需求类型建议：`配置`、`接口`、`规则`、`修复`、`缓存`、`数据`、`其他`。

## 10. 禁止事项

- 不得为同一需求维护两份 PRD。
- 不得创建月份 README 或第二层索引。
- 不得额外创建与需求 README.md 重复的模板文档。
- 不得为了套模板而填写无意义章节。
- 不得擅自脑补业务规则、接口字段或验收口径。
- 不得修改历史需求的已确认口径，除非用户明确要求。
- 不得将明显需要 HTML 原型的需求强行放入本仓库。
- 不得用 GitHub 索引重复维护禅道中的开发、测试和上线状态。
