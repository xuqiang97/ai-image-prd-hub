# AGENTS.md

本文件用于指导 ChatGPT、Codex 及其他 AI 在本仓库中新增和维护 AI 生图轻量需求 PRD。

## 1. 仓库定位

本仓库只收录**不需要单独制作 HTML 原型**的 AI 生图轻量需求。

判断标准：

> 本需求是否需要先通过页面原型确认交互和展示效果。

如答案为“需要”，则不要将需求直接放入本仓库，应建议单独建立原型仓库，通过 `HTML 原型 + README.md PRD` 交付。

## 2. 适合收录的需求

包括但不限于：

- 取数逻辑优化
- 接口字段调整
- 配置项规则
- 状态流转
- 前端 localStorage 等本地缓存规则
- 数据同步
- 后端处理口径
- 定时任务
- 线上问题修复
- 页面无明显变化的轻量前端逻辑

## 3. 新增需求操作流程

收到新需求后，按以下顺序处理：

1. 判断是否需要 HTML 原型。
2. 梳理需求边界、是否拆分和关键缺失信息。
3. 按 `templates/prd-template.md` 生成唯一一份 PRD。
4. 在对应月份下新建需求目录和 `README.md`。
5. 更新仓库根目录 `README.md` 的唯一需求索引。
6. 不创建月份 README，不维护第二份索引，不复制生成钉钉版 PRD。

## 4. 目录规则

正式需求：

```text
prd/YYYY-MM/M-禅道号-英文短名/README.md
```

暂无禅道号：

```text
prd/YYYY-MM/TEMP-YYYYMMDD-英文短名/README.md
```

英文短名规则：

- 使用小写字母和中划线。
- 建议 3～6 个单词。
- 优先描述需求核心，不使用无意义的 `update`、`change`、`optimize` 单独命名。

## 5. 文档唯一性

每个需求目录只维护一份 `README.md`，该文件同时用于：

- GitHub 需求沉淀
- 研发和测试实施参考
- 复制到钉钉需求流程
- 后续 AI 读取和复用

不得再额外生成：

- GitHub 实施版 PRD
- 钉钉精简版 PRD
- 月份汇总版 PRD

## 6. PRD 写作规则

### 6.1 前部格式

前半部分尽量遵循公司钉钉模板：

- 标题
- 禅道需求
- 修改纪要
- 调研纪要
- 业务背景和价值
- 需求概述
- 功能需求
- 风险描述
- 验收标准

### 6.2 后部结构

后半部分按需求实际情况灵活展开，不机械填写所有章节。

可按需增加：

- 数据和接口口径
- 字段定义
- 取数优先级
- 异常和边界
- 历史数据处理
- 服务验收
- 初始化权限
- 测试样例
- 回归范围
- 参考链接

不适用的章节直接删除，不写空洞内容。

### 6.3 小需求写法

小需求应保持轻量，重点写清：

- 为什么要做
- 本期具体改什么
- 哪些范围不改
- 规则和优先级
- 异常情况如何处理
- 如何验收

避免模糊表达，例如：

- 尽量
- 可能
- 视情况
- 正常处理
- 相关逻辑
- 保持一致

应改为可执行、可测试的明确规则。

### 6.4 修改纪要

最终 PRD 通常只保留 `V1.0`，不记录 AI 与用户之间的多轮讨论过程。

### 6.5 调研纪要

只记录有效的业务调研和关键干系人结论。没有调研信息时，可写“暂无”，不要虚构调研对象或结论。

## 7. 常见需求重点

### localStorage / 本地缓存

写清：

- 缓存内容
- 写入时机
- 读取时机
- 默认值
- 账号是否隔离
- 浏览器和设备维度
- 清缓存后的表现
- 是否影响历史任务

### 取数逻辑优化

写清：

- 触发入口
- 查询参数
- 数据源
- 优先级
- 多条命中规则
- 无数据处理
- 单个和批量入口是否统一
- 历史逻辑影响

### 接口字段调整

写清：

- 字段名称和类型
- 新增、修改或废弃
- 是否必填
- 默认值
- 前端传参
- 后端兼容
- 历史数据处理

### 线上问题修复

写清：

- 问题表现
- 影响范围
- 修复口径
- 不影响范围
- 是否补数据
- 验收样例

修复型需求不要写成重 PRD。

## 8. 索引维护

只维护根目录 `README.md` 中的需求索引。

新增需求时补充：

- 日期
- 需求编号
- 业务端
- 需求名称
- 需求类型
- 状态
- PRD 链接

业务端建议：`运营`、`美工`、`公共`。

需求类型建议：`配置`、`接口`、`规则`、`修复`、`缓存`、`数据`、`其他`。

## 9. 禁止事项

- 不得为同一需求维护两份 PRD。
- 不得创建月份 README 或第二层索引。
- 不得为了套模板而填写无意义章节。
- 不得擅自脑补业务规则、接口字段或验收口径。
- 不得修改历史需求的已确认口径，除非用户明确要求。
- 不得将明显需要 HTML 原型的需求强行放入本仓库。
